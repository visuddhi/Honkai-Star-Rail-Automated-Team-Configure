# 推荐与评分说明

这份文档对应当前仓库实现，目标是明确回答两个问题：

1. 推荐是怎么生成的
2. `score` 到底是怎么来的

## 1. 当前推荐流程

代码入口：

- `src/data_loader.py`
- `src/recommender.py`
- `src/simulator.py`

整体流程分三层：

### 第一层：把盒子 JSON 解析成标准角色对象

在 `src/data_loader.py` 里完成。

当前会读取：

- 角色名 / 角色 ID
- 等级
- 星魂
- 行迹
- 光锥
- 遗器总分
- 遗器详细信息（如果你提供了 `relics` / `artifacts` / `equipment` / `ornaments` 等结构）

最终每个角色会被标准化成：

- `id`
- `name`
- `investment`
- `build`
- `base`
- `raw`

其中：

- `investment` 是练度总分
- `build` 是从遗器和面板推出来的派生属性
- `base` 是角色库里的静态模板数据

### 第二层：静态配队推荐

在 `src/recommender.py` 里完成。

当前支持两种场景形态：

- 双关场景：例如混沌 / 虚构 / 末日的双半区
- 单关场景：例如这次新接入的 `异相仲裁` 单个骑士关或王棋关

#### 2.1 先穷举所有 4 人组合

当前不是遗传算法，也不是 MILP，也不是局部搜索。

它做的是：

- 先对当前盒子里所有可识别角色做 `combinations(roster_units, 4)`
- 也就是把所有 4 人队完整枚举一遍

这一点在：

- `src/recommender.py` 里的 `all_combos = list(combinations(roster_units, 4))`

所以严格来说：

- 单队搜索是穷举
- 不是近似

#### 2.2 对每个 4 人队分别算“上半分”和“下半分”

每个队伍会经过：

- 队伍合法性过滤
- trait 供给计算
- 场景匹配打分
- 配队结构加分
- 协同加分
- SP 惩罚
- 练度加成

#### 2.3 再把上半候选和下半候选两两配对

这里目前是近似，不是全局最优求解。

实现是：

- 上半候选按分数排序
- 下半候选按分数排序
- 如果候选规模不大，会做全量精确配对
- 如果候选规模偏大，会回退成 `Top 40 x Top 40` 的近似配对

也就是说：

- 单队是穷举
- 双队配对在小规模时可以精确
- 双队配对在大规模时仍然是近似
- 单关场景则不走配对，直接输出单队 Top 结果

如果你说“现在没有组合优化”，这句话在旧版更成立；当前版本更准确的描述是：

- 有组合搜索
- 单队层是穷举
- 双队层是“能精确则精确，否则回退近似”

### 第三层：对 Top 结果做回合模拟

在 `src/simulator.py` 里完成。

当前流程：

- 静态推荐出 Top 结果
- 只对 Top 3 跑模拟
- 每套抽样 24 次
- 输出 `普通随机 / 最差随机 / 最佳随机`
- 当前模拟已经有：
  - 终结技插队窗口
  - 危险技能 / 蓄力动作
  - 召唤物与额外场面压力
  - 按血线触发的阶段转换

这里的“最差/最佳”不是绝对极限 RNG，而是分位数视角。

## 2. 当前 `score` 怎么算

### 2.1 角色练度分 `investment`

在 `src/data_loader.py` 里算。

当前主要由下面这些组成：

- 等级
- 星魂
- 行迹
- 光锥
- 遗器

大致权重是：

- 等级：34
- 星魂：16
- 遗器：22
- 行迹：18
- 光锥：10

总和约等于 100 分制。

### 2.2 角色 trait

每个角色在 `data/characters.json` 里有一份基础模板：

- `aoe`
- `single`
- `break`
- `debuff`
- `dot`
- `fua`
- `speed`
- `survival`
- `support`
- `sp`

这些不是实战倍率，而是抽象能力标签。

运行时会做两次变换：

1. 用 `investment` 放大基础 trait
2. 再用 `build` 把遗器 / 面板影响叠进去

现在遗器已经不是只看 `relicScore` 了：

- 如果你提供详细遗器结构，会解析主词条 / 副词条 / 套装
- 再映射成 `speed / break / crit / dot / debuff / fua / support / heal / energy / survival / damage`
- 这些会继续反映到最终 trait 上

换句话说，现在已经开始“体现遗器信息”，但还是第一版启发式映射，不是完整词条模拟器。

### 2.3 单队静态分

单队分在 `src/recommender.py::_score_team` 里算。

它由这些部分组成：

- `trait_score`
  - 把队伍 trait 和场景目标值做比对
  - 每个 trait 会乘场景权重
- `synergy_score`
  - 来自角色库手写的协同关系
- `composition_score`
  - 看队伍结构是不是合理
  - 例如单生存、双辅、虚构低生存等
- `element_score`
  - 是否命中场景偏好弱点元素
- `investment_bonus`
  - 全队平均练度带来的小幅奖励
- `penalty`
  - SP 太差
  - support 不足
  - 击破不足
  - 非 sustain 位和场景目标不匹配

最后合成：

```text
单队总分
= trait_score
+ synergy_score
+ composition_score
+ element_score
+ investment_bonus
- penalty
```

### 2.4 双队总分

双队分不是重新完整模拟出来的，而是静态合成：

```text
双队总分
= (上半分 + 下半分) / 2
+ 平衡奖励
```

平衡奖励会偏好：

- 上下半都强
- 而不是一边特别高、一边拖后腿

## 3. 当前模拟器怎么打分

### 混沌 / 末日

当前更像“通关稳定性分”：

- 是否清掉波次
- 等效轮次
- 敌人行动次数
- 破韧次数
- 生存剩余

### 虚构

当前更像“刷分目标分”：

- 击杀数
- 连续击杀滚雪球
- 模拟积分
- 破韧次数
- 生存余量

## 4. 你刚才指出的问题，哪些是对的

### 对的部分

- 之前确实没有真正读取遗器详细结构，只是用了 `relicScore`
- 当前仍然没有做严格的双队全局最优搜索
- 当前角色模型仍然是 trait-based，不是逐技能、逐倍率、逐词条的完整战斗复刻

### 现在已经改进的部分

- 已开始解析详细遗器 / 面板信息
- 遗器会影响静态推荐和回合模拟
- 模拟已经有场景专属波次脚本和部分阵容专属行为
- 模拟已经有第一版终结技插队、危险技能、召唤物和阶段转换

## 5. 当前版本最大的限制

### 还没有完整做到的

- 没有逐技能倍率表
- 没有完整的逐角色能量阈值、终结技动画时点和回能细节
- 没有完整敌方 AI / 点名 / 控制 / 抗性系统
- 没有完整 2+2 / 4 件套 / 位面套精确效果复刻
- 没有把双队配对升级成严格全局优化器
- 还没有把 `异相仲裁` 的三骑士做成“三队同时分配”的全局求解

## 6. 如果下一步继续做，最值得补什么

优先级建议：

1. 把双队配对从“阈值回退近似”继续推进到更稳定的全局最优搜索
2. 把遗器解析继续细化到套装效果和关键主词条
3. 把模拟器继续从“机制级原型”推进到更完整的敌方 AI、抗性和技能脚本
